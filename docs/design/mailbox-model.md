# Mailbox model

Aerogram treats the `Mailbox` entity as autonomous. The decision is structural: it shapes the schema, the SCIM endpoint and the multi-tenant isolation strategy.

## The core entity

```rust
pub struct Mailbox {
    pub id: MailboxId,           // UUID v7 generated by Aerogram
    pub tenant_id: TenantId,     // UUID opaque, sourced from the JWT claim
    pub address: Address,        // RFC 5321 mailbox-local + domain
    pub display_name: Option<String>,
    pub quota_bytes: Option<u64>,
    pub status: MailboxStatus,   // Active, Suspended, Deactivated
    pub external_id: Option<ExternalId>, // SCIM externalId, free-form opaque
    pub created_at: DateTime<Utc>,
    pub updated_at: DateTime<Utc>,
}
```

Three points are deliberate:

1. **Primary key is the pair `(tenant_id, address)`**, not a Nubster Identity Account UUID. Aerogram owns the lifecycle of mailboxes, independently of any external identity store.
2. **`external_id` is a free-form opaque string**. SCIM uses it to correlate a mailbox with the identity record it represents in the operator's IdP. The IdP may be Nubster Identity or any other standards-compliant OIDC provider.
3. **No foreign key to identity tables.** The mailbox row never references `identity.account.id` or any equivalent. Cross-system correlation goes through `external_id` only.

## Why autonomous

The autonomous model is required by the no-lock-in commitment documented in [`interop.md`](interop.md):

- An operator running Aerogram with a third-party IdP must not store anything referencing a Nubster Identity table.
- An operator switching from one IdP to another must do so without rewriting Aerogram's schema.
- An operator using Aerogram standalone (no IdP at all, with locally provisioned tokens) must still be able to create mailboxes.

## How provisioning happens

Three provisioning paths coexist:

### 1. SCIM 2.0 from any IdP (recommended for multi-IdP setups)

```http
POST /scim/v2/Users
Authorization: Bearer <token-with-scim:write-scope>
Content-Type: application/scim+json

{
  "schemas": ["urn:ietf:params:scim:schemas:core:2.0:User"],
  "userName": "alice@example.org",
  "externalId": "idp-uuid-of-alice",
  "active": true,
  "name": { "givenName": "Alice", "familyName": "Doe" }
}
```

Aerogram creates a `Mailbox` row with `tenant_id` resolved from the JWT, `address` resolved from `userName`, `external_id` set to the SCIM `externalId`.

### 2. CLI by the operator (for self-hosted bootstrapping)

```sh
aerogram mailbox create \
  --tenant 4f0b... \
  --address alice@example.org \
  --display-name "Alice Doe"
```

No identity provider involved. `external_id` is left `None`.

### 3. Admin SSR (for operators who prefer a UI)

The admin dashboard exposes a form that maps onto the same SCIM endpoint internally.

## The optional Identity view

Nubster Identity may keep a view that joins its `Account` table with Aerogram's `Mailbox` table for convenience (for example to render an account profile with the user's mailbox address). The view lives **on the Identity side**:

```sql
-- In the Nubster Identity database (not in Aerogram's database)
CREATE OR REPLACE VIEW identity.account_mailbox AS
  SELECT a.id           AS account_id,
         a.subject_id   AS subject_id,
         a.handle       AS handle,
         m.address      AS mailbox_address,
         m.tenant_id    AS mailbox_tenant_id
  FROM identity.account a
  LEFT JOIN aerogram.mailbox m
    ON m.external_id = a.id::text  -- if Identity provisions via SCIM
    OR m.address = (a.handle || '@' || a.domain);
```

This view is a convenience for Identity-side dashboards. It is not part of Aerogram's API surface and is not assumed by any Aerogram code path.

## Multi-tenant isolation

`tenant_id` is the only isolation key. Every query against `aerogram-store`, `aerogram-queue`, `aerogram-search` and `aerogram-spam` is parameterised by `tenant_id`. The JWT claim that resolves the tenant is configurable:

```toml
[auth]
tenant_claim = "tenant_id"   # default
# Alternatives the operator may pick:
# tenant_claim = "org"
# tenant_claim = "https://example.org/tenant"
```

Aerogram never trusts a `tenant_id` provided in a request body or query parameter. It always comes from a verified JWT claim or from the operator-side CLI.

## Storage layout

```
aerogram.mailbox             -- the entity defined above
aerogram.mailbox_alias       -- additional addresses pointing at the same mailbox
aerogram.mailbox_forwarding  -- forwarding rules (RFC 5321 4.4)
aerogram.mailbox_sieve       -- v0.3+, per-mailbox Sieve filters
```

All four tables include `tenant_id` as part of their primary key or as a `NOT NULL` foreign key into `aerogram.mailbox`.

## Migrations

The schema lives under `crates/aerogram-store/migrations/`. Each migration file follows the `Vxxxx__name.sql` convention. The CLI applies them through `aerogram migrate`; the runtime never auto-applies migrations on start-up.
